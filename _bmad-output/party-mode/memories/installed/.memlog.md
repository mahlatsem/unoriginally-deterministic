- (moment) First session. Debated VS Code alternatives for the toolkit: (1) Claude Code skill + plain CLI scripts, no extension; (2) MCP server exposing primitives + thin rendering companion; (3) standard-format output (Mermaid/DOT) via VS Code's built-in previewers. Sally kept forcing the room back to FR-4 (click-to-source) as the real constraint every option has to survive. Winston leans MCP+thin-webview as most durable. Amelia: a minimal click-to-source webview is a weekend, not a quarter - the rest can live outside the extension. Paige connected skill+scripts to the earlier GT-book/self-documentation (OQ-5) discussion. Mary frames low-friction distribution as the sharper anti-GT wedge than a bespoke extension.
- (outcome) LOCKED: v1 delivery shape is skill + CLI scripts + a minimal pixel-perfect-clickable webview (not a full custom VS Code extension), with primitives exposed as a small well-typed TypeScript library (not MCP). Key turn: user clarified the molding AI's real job is learning the toolkit's own libraries (tree-sitter/primitives) fast, not reasoning about the target codebase - this flipped Winston off his MCP-forward lean, since a plain typed library is the well-trodden 'AI reads API+examples, writes calling code' pattern, and it keeps FR-3's audit-the-source trust check as the natural default (a persisted script) rather than an opt-in extra step over live MCP tool-calls. MCP demoted to a possible post-OQ-1 reach layer, not a v1 decision. This is architecture-level, not a PRD FR change - John flagged it belongs in addendum/architecture notes, not a PRD rewrite.
- (outcome) Session closed. Full architecture thread resolved: v1 delivery = thin VS Code extension (webview+file-watcher only) + bundled starter-tool scripts + molded tools in user's repo, primitives as a plain TS library (MCP deferred post-OQ-1), LSP-based parked as a possible v2 complementary layer (native hover/CodeLens/diagnostics) but rejected as primary architecture since it can't do whole-system graphs or bespoke molding. Commons-graduation mechanism defined: molded tool -> upstream PR -> becomes bundled starter tool in a future extension release. All captured in the PRD workspace's addendum.md, not the PRD itself, per John's repeated flag that this is architecture-shaped.
