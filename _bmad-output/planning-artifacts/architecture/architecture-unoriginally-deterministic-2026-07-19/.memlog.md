- (decision) Working mode: Coaching path. Deliverable: spine only (sole builder, no alignment artifact needed).
- (decision) PARADIGM: functional core / imperative shell. Core = extraction and every perspective are pure functions over an immutable snapshot (parse(files)->Snapshot; perspective(snapshot,scope)->ViewDoc); shell = extension does all I/O (file watch, webview messaging, editor jumps). Binds: no hidden state or I/O inside primitives or molded tools. Prevents: determinism-by-discipline divergence — 'same code, same structure' becomes true by construction; molded tool = one pure function, the narrowing that maximizes gate odds. Rejected knowingly: GT-style live moldable objects w/ attached views — richer but drags in lifecycle + mutable session state, exactly where independently-built tools diverge.
- (decision) CONTRACT IS TWO LAYERS, provenance enforced at BOTH (option C). Layer 1 = typed TS library (tree-sitter query prims + drawing prims); builder API draw.node(span,...)/draw.edge(span,...) — span required by function signature; this is what Claude Code learns and the gate tests. Layer 2 = ViewDoc JSON document format, folder-watched, renderer blind to producer; schema-validated at the render boundary — spanless elements rejected even in JSON the library never saw. Prevents: bypass of FR-4 via hand-written/foreign JSON; guarantee attaches to the product, not the happy path. ViewDoc schema is versioned — it is the FR-6 stability contract molded tools frozen in user repos depend on.
- (decision) EXECUTION MODEL confirmed as invariant: library runs in PLAIN NODE with zero VS Code dependency; a molded tool is a standalone Node script, never an extension plugin; no primitive may reach into the editor. Starter perspectives run in-process in the extension command handler; molded tools run user-invoked (terminal/npm/Claude Code); both emit ViewDoc to the conventional output folder. Load-bearing for FR-6 durability and for the gate protocol (tool runnable from terminal without extension-host harness).
- (decision) NODE IDENTITY = canonical typed URI, minted by a contract-provided function, never hand-assembled by tool authors. Shape: <lang>:<kind>/<canonical-name> e.g. cobol:program/PAYROLL, cobol:table/TAX_RATES, cobol:file/EMPFILE, cobol:paragraph/PAYROLL#3000-TAX-DISPATCH. Binds: every producer of a node or edge endpoint. Prevents: THE divergence that would silently kill inbound reach — two tools naming the same table differently means 'what writes to TAX_RATES' returns a partial answer that LOOKS complete. Highest-value invariant in the spine; also what makes molded tools composable and cross-referenceable at all.
- (decision) SNAPSHOT: single owner, canonically ordered, content-addressed. One extraction service yields an immutable Snapshot; EVERY collection inside is sorted by a defined total order before any consumer sees it; snapshot identity = content hash of (grammar versions + canonicalized file set + extraction version). Binds: extraction and all perspectives. Prevents: (a) divergent snapshots from per-tool extraction, (b) ordering-derived layout nondeterminism at its source, (c) makes the UX spine's 'stated snapshot / drawn at HH:MM' literally true and lets staleness be detected by hash comparison rather than timestamps.
- (decision) DETERMINISM RESOLVED IN FAVOUR OF THE STRONGER CLAIM (resolves the fork PRD s4 handed architecture). Layout is deterministic by construction: canonical input ordering (from the Snapshot) + a LAYERED, non-force-directed engine (no RNG, no seed) + pinned engine version. Consequence: product copy may claim 'same code, same PICTURE' and a rendered view becomes auditable evidence — restoring the PRFAQ's reproducibility promise that the PRD had narrowed to structure-only. UPSTREAM: PRD s4 non-goal 'Not pixel-stable view layout at v1' must be updated — architecture took the other branch. Research note: dagre and ELK.js are both layered/non-force-directed so neither carries seed risk, but NEITHER documents a determinism guarantee — engine choice deferred to implementation, the INVARIANT (canonical order + no force-directed + pinned version) is what binds.
- (decision) DATA-ARTIFACT EXTRACTION = two-stage parse, fixed as the pattern. Stage 1: COBOL grammar locates EXEC blocks and file declarations (SELECT/ASSIGN/FD + I-O verbs). Stage 2: a second parser reads EXEC block contents into the SAME node/edge model via the canonical-URI minter. Binds: any language profile whose grammar embeds another language. Prevents: each profile inventing its own embedded-language handling, and data lineage silently omitting SQL-touched tables. Verified practicable: EXEC-block mining for CICS/SQL targets is an established technique and tree-sitter-sql exists; specific SQL grammar left to implementation. Unreadable EXEC block degrades to an unparsed region (existing honest-state machinery) — visible partial coverage, never silent.

- (decision) LAYOUT PLACEMENT: tools emit TOPOLOGY ONLY; the pipeline lays out. A molded tool emits nodes/edges and never touches layout — the smallest possible tool surface, which is what SM-2's odds rest on. Layout is a pure deterministic function applied downstream; the rendered artifact records layout engine + version so the picture is reproducible without tools knowing layout exists. Binds: every producer. Prevents: two tools emitting differently-shaped documents for the same job, and contract growth before the gate.
- (decision) MOLDED-VIEW INTERACTIVITY is document-scoped, and stated plainly rather than faked. A molded view supports every interaction its ViewDoc can serve (provenance, click-to-source, selection). Drilling into a grain the tool did not emit requires re-running the tool, and the UI SAYS SO rather than offering a dead affordance. Binds: renderer + molded tools. Prevents: the renderer needing to know its producer (preserves the addendum's blind-renderer decision), and the extension executing arbitrary user code on navigation. Accepted cost: bundled perspectives are more interactive than molded ones — an asymmetry made visible, consistent with FR-7 already showing the bundled-vs-molded update asymmetry.
- (decision) DISTRIBUTION: published npm package; the extension bundles the same library for bundled perspectives. One artifact, two consumers. Satisfies FR-6 — npm is not a TOOLKIT-HOSTED registry (the consequence's actual wording) and published versions are effectively immutable, so a molded tool outlives the project. Also supplies the lockfile-pinned versioning the determinism claim needs, since layout-engine version is part of reproducibility. Rejected: extension-provided primitives (conflicts with FR-6's explicit 'no marketplace call' consequence); vendoring (strongest independence but no upgrade path, tools frozen at copy time).
- (decision) DETERMINISM VERIFICATION: golden-file tests as a RELEASE GATE. Committed fixture repo -> expected ViewDoc and expected laid-out output, byte-compared in CI; drift in grammar version, canonical ordering, or layout engine fails the build. Binds: release process. Prevents: the determinism claim decaying into an aspiration — makes it falsifiable, which is the only version of the claim this product is entitled to make.
- (decision) EXCHANGE CONVENTION: one conventional, workspace-relative output folder holds ViewDocs; the extension watches it and renders whatever valid ViewDoc appears, blind to producer (ratifies the addendum's blind-renderer decision). Snapshot cache is separate from ViewDoc output, is a derived artifact, and is git-ignored by default — it is reproducible from source by definition, so committing it would invite a stale cache being trusted as evidence. Binds: bundled perspectives, molded tools, extension. Prevents: producers writing to divergent locations, and cache being mistaken for output.
