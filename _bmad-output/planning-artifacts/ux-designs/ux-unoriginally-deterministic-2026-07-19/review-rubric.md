# Spine Pair Review — unoriginally-deterministic

Reviewed: `DESIGN.md` + `EXPERIENCE.md` (both `status: draft`, updated 2026-07-19) against `design-md-spec.md`, the three DESIGN examples, the two EXPERIENCE examples, PRD (2026-07-18), PRFAQ, PRFAQ distillate, and the workspace `.memlog.md`.

## Overall verdict

A downstream consumer can source-extract this pair cleanly: every `{path.to.token}` reference in both files resolves to a defined frontmatter token, all three frontmatter sources exist, PRD/PRFAQ citations are verbatim and accurate, and the three invented sections carry the product's actual load. Two high-severity issues stand between this and a safe architecture handoff: (1) the spine supersedes PRD OQ-3 and materially expands FR-5 (one starter view → six perspectives across three grains), but that supersession is recorded only in `.memlog.md` — a consumer reading PRD + spines sees an unflagged contradiction with no declared winner; (2) the theme-inheritance posture leans on `--vscode-charts-*` tokens with no fallback chain or contrast floor, so the signature provenance tick can silently vanish in themes that don't define chart colors. Below those, a consistent band of medium gaps: secondary visuals (view header, category legend, GO TO/sequence marks, gallery) lack specs, and two canvas states (nothing-parseable scope, stale-open-view-after-edit) are missing.

## 2. Token completeness — adequate

Extracted every frontmatter token (19 colors incl. `cat-1..6`, 4 typography ramps, 4 rounded, 6 spacing, 11 components) and every `{path.to.token}` reference in the prose of both files plus the component objects. **All references resolve** — including EXPERIENCE.md's six token references (`{components.provenance-tick}`, `{components.node-unparsed}`, `{components.edge-traced}`, `{components.anchor-popover}`, `{components.provenance-rail}`, `{colors.trace-active}` ×2, `{colors.focus-ring}`), which resolve to DESIGN.md tokens by name. The `var(--vscode-*)` posture is a legitimate platform-convention analogue of the spec's UI-system-inheritance pattern, and it is applied consistently: no hex anywhere, base tokens routed through the `colors` map, deltas only.

### Findings
- **high** No fallback chain or contrast floor for the semantic brand layer. `provenance`, `cat-1..6` bind to `--vscode-charts-*` (DESIGN.md:23, 28–33), which not every theme (especially custom/older themes) defines; an undefined CSS custom property resolves to nothing and the provenance tick — the motif whose "absence is impossible by construction" (DESIGN.md:187) — becomes invisible, a silent trust-model break. The file asserts contrast is "inherited, never hand-tuned" (DESIGN.md:21–22) but states no minimum for the load-bearing combinations (tick on `node-surface`, `unparsed` text on `node-surface`, `trace-active` on `selection`). *Fix:* declare CSS fallback chains for every `charts-*` binding (e.g. `var(--vscode-charts-green, var(--vscode-testing-iconPassed, currentColor))`), plus a stated floor: ≥3:1 non-text contrast against `node-surface` for tick/edge colors, ≥4.5:1 for `unparsed` label text, and a minimum tick size so the mark survives low-contrast themes.
- **low** `anchor-popover` (DESIGN.md:98–104) is the only component whose color values bypass the `colors` map (raw `var(--vscode-editorHoverWidget-*)`). A consumer extracting the `colors` object won't see the hover-widget family. *Fix:* lift the three hover-widget values into `colors` (`popover-background` etc.) and reference them.
- **low** "`{spacing.1}`–`{spacing.8}`" (DESIGN.md:163) implies a continuous scale; `spacing.5` and `spacing.7` are undefined (scale is 1,2,3,4,6,8). *Fix:* write "the `{spacing.1}`…`{spacing.8}` scale (4/8/12/16/24/32)".

## 3. Component coverage — adequate

Extracted every component name used anywhere in both spines and cross-checked DESIGN.md.Components (visual) against EXPERIENCE.md.Component Patterns (behavioral). Fully paired, with real rules on both sides: Node, Unparsed node, Edge, Provenance tick, Source anchor, Anchor popover, Provenance rail, Perspective tabs/switcher, Breadcrumb. Behavioral rows are genuine rules (dismissal ladders, sync directions, invariants), not one-word descriptions.

### Findings
- **medium** "View header" has a behavioral row (EXPERIENCE.md:69) but no DESIGN.md.Components spec — its visual treatment exists only as "Legend/controls" inside Layout & Spacing (DESIGN.md:168), under a different name. Downstream extracting DESIGN.md.Components will miss the top strip entirely. *Fix:* add a `view-header` component (frontmatter + Components bullet) and use one name in both files.
- **medium** The category legend is load-bearing for accessibility ("never color alone" — EXPERIENCE.md:64, 111; DESIGN.md:147, 205) yet has no visual spec and no behavioral row anywhere. Is it always visible? Collapsible? Does it filter on click? *Fix:* add a legend row to Component Patterns and a legend entry under DESIGN.md.Components.
- **medium** Required visual distinctions with no visual spec: `GO TO` edges "visually distinguished" (EXPERIENCE.md:137–138), cycles "marked explicitly" (137), sequence guard labels and `PERFORM UNTIL` loop fragments (135). DESIGN.md:193 says only that sequence/flowchart marks "reuse `{components.node}` / `{components.edge}` tokens" — the distinguishing treatment is left for the implementer to invent, on marks the flows call out as the payoff (Flow 2 step 4). *Fix:* one line each in DESIGN.md.Components: e.g. `GO TO` edge = dashed stroke + distinct arrowhead; cycle badge; guard label set in `{typography.code}` on the message.
- **medium** View gallery / Gallery item (`[ASSUMPTION]`, EXPERIENCE.md:31, 72) has behavioral rules but no visual spec and no decision on surface type — native QuickPick (no DESIGN spec needed, per the inherit-VS-Code contract) vs. webview (needs one). The ambiguity is acknowledged in memlog line 15 but the spine doesn't state which way to default. *Fix:* commit the default (QuickPick is the low-cost, posture-consistent choice) and tag it `[ASSUMPTION]`.
- **low** Naming variance: DESIGN.md "Perspective tabs" / token `perspective-tab` vs. EXPERIENCE.md "Perspective switcher"; DESIGN.md "Node — unparsed" vs. EXPERIENCE.md "Unparsed node". Resolvable by a human, noise for mechanical extraction. *Fix:* pick one name per component and use it verbatim in both files.

## 4. State coverage — adequate

Walked all eight IA surfaces (EXPERIENCE.md:28–38) against State Patterns (76–84) plus states embedded in Component Patterns, Interaction Primitives, and Responsive & Platform. Covered: canvas extracting / honest holes / fully-unparseable file / re-run / molded-tool run error; gallery empty; editor trace target; popover open/pin/dismiss; large-graph and narrow-panel behavior. The "no AI spinner" and "never blank canvas" treatments are exactly the right invariants.

### Findings
- **medium** No state for a scope with *nothing parseable*: user runs "Open dependency map" on a folder/repo with no COBOL (or no supported sources) at all. "Fully unparseable file" (EXPERIENCE.md:80) covers one file that parses to all-ERROR, not an empty extraction result. Downstream will invent something — and a blank canvas here would violate the spine's own rule. *Fix:* add a row: "Nothing to draw — 'No COBOL sources found under ⟨scope⟩. Nothing here is guessed.' + pointer to supported languages."
- **medium** No state for a stale open view after the user edits code. Views re-derive per run (EXPERIENCE.md:83), but while a view stays open (Flow 1 step 5: "she keeps the view open") the code can change underneath it, and a `file:line` anchor click can then land on the wrong line — the trust destination lying. The header's "drawn from … and when" (EXPERIENCE.md:69) mitigates but no treatment defines whether anchors re-validate at click time or the view marks itself stale. *Fix:* add a row: on jump-to-source where the file changed since draw, state it plainly ("drawn 10:04, file changed since — re-run to redraw") rather than silently landing.
- **low** Sequence perspective "asks for one [entry point] if nothing is selected" (EXPERIENCE.md:70, 135) — the ask's treatment (QuickPick? inline prompt?) is unspecified. *Fix:* name the surface (native QuickPick fits the posture).
- **low** Molded-tool *discovery/registration* failure (malformed tool, invalid perspective declaration) is uncovered — only run-time error is (EXPERIENCE.md:84). *Fix:* one row: malformed tool listed in gallery as unloadable with its error and `file:line`, never silently dropped.
