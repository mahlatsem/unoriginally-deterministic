- (change) GAP FOUND + CLOSED (user question, 2026-07-19): inbound navigation was missing entirely — 'called by / callers / inbound / dependents / reverse / fan-in' appeared NOWHERE in either spine or the PRD. Every specified perspective pointed outward (reach tracer 'reads, writes, invokes'; sequence 'from a chosen entry point'; dependency-map edges 'one unit naming another'). Notable: the PRFAQ's original molding example DID include an inbound query ('show me where the Java invokes it') — it was cut with the Java non-goal and the inbound direction was never separately preserved. Also a framing mismatch: the PRD called the reach tracer 'the blast-radius perspective' while specifying only the outbound direction, but blast radius for changing X is predominantly who depends on X. DECISION: make the reach tracer BIDIRECTIONAL (direction toggle: outbound / inbound / both) rather than adding a seventh perspective — reuses its depth-truncation discipline and honest-state vocabulary.
- (constraint) INBOUND COMPLETENESS is a strictly harder honesty problem than outbound, and is now a hard rule. Outbound: 'X invokes something unresolvable' — the unknown is visible and the existing unresolved-target state covers it. Inbound: 'nothing reaches X' can be FALSE, because any dynamic-dispatch site in the estate might resolve to X at runtime. So an inbound result is NEVER a bare count or a bare empty state: it always states the unresolvable-dispatch population that could reach the target ('4 static callers, plus 12 dynamic call sites whose targets are not statically resolvable — any could reach this'). An empty inbound result must never read as 'unused' or 'safe to delete' — that false negative is precisely the confidently-wrong, business-breaking failure the product exists to refuse. Banned: any disposal/dead-code affordance derived from inbound emptiness.
- (decision) DATA ARTIFACTS BECOME FIRST-CLASS NODES (user, 2026-07-19). Driver: the real questions are 'what depends on this DB table / file / program' (inbound) and 'once this component is invoked, what else does it touch' (outbound). Direction alone did not answer the first — the thing the user wants to SELECT was not a node. System grain admitted only compilation units; files appeared once as an edge-label example; DB tables and EXEC SQL appeared NOWHERE in either document (verified by grep). Also fixed an incoherence introduced with the bidirectional tracer: it claimed first-class read/write edge kinds while no node type existed on the far end of a write. RESOLUTION: system grain now admits programs, copybooks, FILES and TABLES. Data artifacts are LEAF nodes — no component grain, so drilling is unavailable (availability rule covers it) and the natural action on one is inbound reach.
- (constraint) EXEC SQL / embedded-SQL extraction mechanism is explicitly ARCHITECTURE'S CALL, not a PRD concern (user: 'this is detail for architecture & implementation side'). PRD states only the capability — table- and file-level lineage traceable in both directions. Known risk handed to architecture via addendum: embedded SQL is a second language inside COBOL and the base tree-sitter grammar may treat EXEC SQL...END-EXEC as an opaque blob or ERROR region, so table-level lineage may need a grammar extension or a narrow secondary parse. Not resolved here; flagged so architecture does not discover it late.
