# Adversarial Review — PRD: unoriginally-deterministic (2026-07-18)

- **FR-2's "nothing leaves the machine" claim is false as written.** FR-2 states "extraction, molding, and running a tool all happen locally... no source code or other repository content is ever transmitted off the user's machine," lumping "molding" in with the two genuinely local steps. But molding is explicitly done via Claude Code (§1, Glossary, throughout) — a cloud-hosted AI assistant that necessarily sends the user's source code to a remote API as part of generating a molded tool. As stated, this consequence is either wrong or needs an explicit carve-out distinguishing "analysis-run time" (correctly local, per FR-3) from "molding time" (not local at all).

- **The PRD's own preferred showcase artifact can't be built by v1 as scoped.** OQ-5's leading candidate for the public showcase (SM-3) is "self-documentation" — running a molded tool against unoriginally-deterministic's own source, which is TypeScript. But FR-1 scopes v1 extraction to COBOL only, with no TypeScript extraction capability specified anywhere in §4. The PRD's strongest candidate for its own headline showcase is not producible by the feature set it defines.

- **The real motivating problem (COBOL entangled with Java) sits outside what v1 can actually see, and the provenance mechanism can't tell anyone that.** The Vision/JTBD framing (and the distillate) describe the maintainer's actual reality as COBOL modernization work entangled with Java, but v1 extraction is COBOL-only. Under FR-3's "no source, no node" rule, a call into Java code simply produces no edge — which is visually indistinguishable from "this paragraph has no external callers." In the exact scenario the project exists to solve, the dependency map could look complete while silently omitting a whole class of real dependencies, undermining the trust claim the entire toolkit is built around.

- **"No source, no node" has no accompanying diagnostic, so a tool bug and a true negative look identical.** FR-3 is presented as the structural enforcement of trust, but nothing in the PRD requires the toolkit to report "N elements were skipped due to missing provenance" or equivalent. A molded tool that forgets to pass provenance for some code path is indistinguishable, from the rendered view alone, from a tool correctly reporting that nothing exists there. The trust mechanism prevents fabrication but does not prevent silent incompleteness from being mistaken for verified absence.

- **SM-1 undercuts its own rigor.** It's phrased as "measurably speeds or clarifies comprehension" and then immediately qualified as "self-assessed, no formal instrument" in the same sentence. "Measurably" and "no formal instrument" are in tension — as written this is not a falsifiable success criterion, it's an impression dressed in the language of measurement.

- **SM-2, the single blocking go/no-go gate for the whole project, has no defined pass bar.** "Across several independent tasks" specifies neither a target count nor what makes tasks "independent" (different view types? different COBOL files? different query shapes?), and there is no stated pass-rate threshold for "reliably produce correct... tools." A gate that decides whether any broad build proceeds should not rely on an undefined word ("several") and an undefined success ratio.

- **Several requirements marked "testable" aren't actually operationalized.** FR-2's "the primitive API surface is small enough that its correct usage is the path of least resistance," FR-2's "a molded tool's own source is short enough to be read end-to-end," and the Trust Checks glossary entry's "short, bounded" reading are all asserted under a "Consequences (testable)" heading but carry no quantified threshold — no LOC limit, no complexity bound, no time-to-read budget. Nothing in the document lets a reviewer determine pass or fail on these.

- **It's unclear whether a "starter view" is a shipped, ready-to-run artifact or something that still needs to be built.** §4.3 describes starter views as "hardcoded view types... shipped with the toolkit as a starting point," implying they work out of the box. FR-4 then says the maintainer can "generate and run" a starter view, and separately promises it "works immediately on install" with "no setup" — "generate" implies an authoring/molding step, which contradicts "no setup, works immediately." The PRD uses language for two different delivery models without resolving which one is true.

- **The PRD's own headline design goal — equal ease of use for AI and humans — is never validated by anything in §7.** §4.2 calls the extension contract "the single most important piece of design work in the project" and requires it be "equally easy to build against for AI and for a human reading the API cold," but every success metric and the only reliability experiment (SM-2/OQ-1) test Claude Code exclusively. No metric, consequence, or acceptance criterion validates the human-authoring path at all.

- **FR-1's extensibility consequence is unfalsifiable within v1's own scope.** "Not hardcoded to COBOL-only assumptions that would block adding a second language later" is listed as a testable consequence, but v1 builds no second language. There is no test that can actually run against this claim until someone attempts a second language — at which point any design flaw is already expensive to fix, not cheaply caught.

- **No contingency exists for OQ-1/SM-2 returning "no-go."** The reliability gate is described as one that "must resolve before any broad build proceeds" — the single highest-stakes unknown in the document — yet there is no stated plan for what happens to the contract design, the roadmap, or the project if Claude Code fails to reliably pass both trust checks. For the most consequential open question in the PRD, the document only describes the test, not what follows a failing result.

- **The v1/rollout boundary is ambiguous.** §5 states community contributions open "later, not at v1 launch." §6.2 states contributions open only "after public listing," where public listing is rollout phase 3 (after dogfood and invited beta). It is not clear from the document whether "public listing" is itself inside v1 or a post-v1 milestone, which makes "v1 launch" an undefined point — dogfood-complete? invited-beta-complete? publicly listed on the Marketplace?

- **The choice of COBOL tree-sitter grammar/fork is an unresolved, near-term implementation blocker with no decision path in the PRD.** FR-1 says "base tree-sitter grammar + community extensions," but multiple divergent community forks exist (the distillate names at least five). The PRD makes no call on which fork(s) are adopted, how compatibility or dialect coverage is evaluated, or what happens if the selected grammar can't parse constructs actually present in the maintainer's real legacy code.

- **The user-supplied "heavy dialect parser" extension point is asserted in prose with no spec behind it.** §4.1's Out of Scope carves out commercial/heavy COBOL dialect parsing as "available only as a user-supplied plug-in via an extension point," but no FR, consequence, or acceptance criterion defines or requires this extension point to exist. As written it is a promise, not a requirement — nothing in the document could be used to verify it's actually built.

- **"Never goes stale" and "no re-molding required" assume molded tools survive code evolution gracefully, but the mechanism described doesn't guarantee that.** FR-5 and the Vision claim a molded tool re-derives correctly after code changes with no re-molding step. Neither addresses what happens when a code change breaks the assumptions behind a tool's tree-sitter queries (renamed paragraphs, restructured divisions, etc.) — under FR-3's silent-omission rule, this produces a quietly incomplete view rather than any visible signal that the tool needs attention, which is a durability claim not actually backed by the mechanism as specified.

- **Terminology drift between "starter tools" and "starter views."** §1 (Vision) describes the toolkit shipping "a handful of starter tools," while §4.3 and the Glossary only define "Starter view." It's unclear whether these are the same concept or whether "starter tools" implies additional non-view tooling that is never otherwise mentioned or scoped.

- **The document's own "Working title — confirm" flag is never tracked anywhere.** Line 9 raises an open naming question, but it appears in neither §8 Open Questions nor §9 Assumptions Index — the one open item the document itself calls out is the one that falls through its own tracking mechanism.

- **UJ-3 (showcasing outcomes publicly) has no corresponding requirement.** Nothing in §4 guarantees the toolkit's output is shareable or presentable in any form (export, screenshot-friendly layout, static snapshot, etc.). The journey is realized purely by an implicit assumption that a VS Code screenshot will be good enough, with no feature or acceptance criterion backing it.
