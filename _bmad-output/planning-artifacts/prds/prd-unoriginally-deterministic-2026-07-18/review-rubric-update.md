# PRD Quality Review — unoriginally-deterministic (post-UX-reconciliation update)

*Scope: the 2026-07-19 update reconciling `prd.md` with the finalized UX spines. Reviewed against `.claude/skills/bmad-prd/assets/prd-validation-checklist.md`, with `addendum.md`, `.memlog.md`, and `ux-designs/ux-unoriginally-deterministic-2026-07-19/{DESIGN,EXPERIENCE}.md` as source-extraction inputs.*

## Overall verdict

The update is substantively good work: OQ-3 is closed honestly (struck, not deleted, with the supersession noted), FR-1's three honesty states are a real strengthening of the trust thesis, and §5.2's gate-first phasing is exactly the right instinct — it protects the PRD's own stated bet. What did not survive the edit is the *closure* discipline: §5.2 introduces four deliverables that have no FR anywhere in §3, the new vocabulary (`scope`, `perspective`, `grain`, `extraction snapshot`, view gallery, view trail, provenance rail) never reached §9 Glossary, and §5.4's "nothing additional here" is now false.

The riskiest consequence is boundary ambiguity: §5.1 puts all six perspectives In Scope for MVP while §5.2 conditions Phase A remainder and Phase B on a gate whose failure branch is undefined — so an architect cannot tell what v1 *is*. Combined with "Phase A" now meaning two different bundles in the PRD and the UX spine, this update is not yet safe to hand to epics without a short closure pass.

---

## 1. Decision-readiness — adequate

The gate is stated as a decision, not a consideration, and §5.2's rationale ("a failed gate should not have been preceded by six renderers") names what is being given up. The maintainer's override of the UX spine's two-perspective Phase A down to one is a real decision recorded in `.memlog.md` (line 23) and correctly reflected in the gate-slice row. OQ-3's resolution note is model behaviour — it says *who* resolved it and that the disposition changed.

What is missing is the other half of a gate: the no-go branch. §1 Vision says "Nothing else in this document proceeds until that's answered" and §5.2 labels both remaining tiers "(post-gate)", but nothing anywhere states what happens if SM-2 comes back 2-of-5. Does the toolkit ship Phase A by hand-authored starter tools with the molding thesis abandoned? Does v1 stop? This is the single most consequential decision in the PRD and it has no stated branch — which also means §5.2 currently reads as "everything is post-gate" while §5.1 reads as "everything is in scope," and the reader must reconcile them.

### Findings
- **high** No gate-failure branch defined (§5.2, cross-ref §1, §7 OQ-1) — the PRD gates its entire surface on OQ-1 but never says what v1 becomes if the gate fails. Note that FR-5's own value proposition ("renders on an unmolded repo … no AI/tokens spent") is *independent* of whether Claude Code can mold tools, so a no-go does not obviously kill Phase A. *Fix:* add two sentences to §5.2 stating the no-go branch explicitly (most likely: Phase A ships as bundled starter perspectives, the molding contract is reduced to a hand-authoring contract, SM-2 becomes a non-goal), or add it as OQ-7 if genuinely undecided.
- **high** Phase B's v1 membership is undetermined (§5.1 vs §5.2) — §5.1 lists "Six perspectives across three grains … delivered in the phases below" as MVP In Scope; §5.2 then splits them across a gate and calls Phase A "a complete, honest product." Nothing says whether Phase B is inside v1 or is v1.1/v2. Epics would have to guess the release boundary. *Fix:* state the v1 line explicitly in §5.2 — either "v1 = gate slice + Phase A remainder; Phase B is post-v1 roadmap" or "all three tiers are v1, sequenced" — and make §5.1's bullet agree.
- **medium** The gate validates a contract the PRD says is not the final contract (§5.2, §6 SM-2, §7 OQ-2) — the gate slice deliberately runs against the minimal contract (draw primitives + provenance), while molded-tool perspective registration is a post-gate *contract extension* (Phase B; UX spine says so explicitly). So a 4-of-5 pass answers OQ-1 for a contract that will subsequently grow. The PRD does not acknowledge that the answer may not transfer. *Fix:* one line under SM-2 or OQ-1 stating that the gate result is scoped to the minimal contract and that post-gate contract extensions carry a re-validation obligation.

## 2. Substance over theater — strong

No persona theater (three single-sentence UJs, correctly dialled for a solo project), no NFR boilerplate, and the counter-metrics do real work — SM-C2 ("view-type or language count is never traded against provenance correctness") is now *more* load-bearing after the update, since FR-5 went from one view to six. The Vision paragraph could not be swapped into another PRD; "LLMs are coherence engines, not truth engines" is the actual thesis and the FRs follow from it.

The new FR-1 consequences are the opposite of furniture. "Conflating the two would collapse two different kinds of not-knowing into one vague one" is a real design argument carried into a requirement. No findings.

## 3. Strategic coherence — adequate

The thesis survives the update intact and the phasing is genuinely thesis-driven (it borrows the addendum's MCP-deferral principle by name). Prioritization follows the bet rather than ease.

Two coherence gaps opened. First, the gate slice as written exercises *no* perspective switch and *no* grain drill — §5.2 itself concedes flowchart "is the second perspective and the one that proves the grain model and switcher," and it sits post-gate. So the model that §3.3 commits in full is unproven at the moment the go/no-go fires. That may be the right trade, but it is not named as one. Second, success-metric coverage did not move with scope: the update's largest new build risks — copybook expansion and dual-anchor provenance correctness — validate against nothing. SM-1 validates FR-5/FR-6, SM-2 validates FR-3/FR-4, SM-3 validates FR-5. FR-1 and FR-2 are validated by no SM at all, and FR-1 just absorbed three new consequences.

### Findings
- **medium** No SM validates FR-1 (§6) — copybook expansion, dual-anchor provenance, and the unresolved-target state are now the highest-risk correctness surface in v1 and have no metric. A dual-anchor bug is exactly the "click-through lands on the wrong line" failure FR-1 says would damage trust more than not expanding at all. *Fix:* extend SM-2's "trace to source" pass criterion to explicitly include inclusion-derived elements, or add an SM covering anchor correctness on a copybook-saturated program.
- **low** The gate slice proves neither switcher nor grain model (§5.2) — a defensible trade, but unstated. *Fix:* one clause in the gate-slice rationale acknowledging it.

## 4. Done-ness clarity — thin

This is where the update lost the most ground, and it is the dimension epics lean on hardest.

FR-5's expansion is *scoped* well — the grain/perspective table is precise and language-agnostic, and deferring behavioural detail to `EXPERIENCE.md` is correct. But two of its five new consequences are not testable as written: *"A perspective is offered only where it can honestly derive something for the current scope and grain"* has no predicate an engineer can implement (the UX spine says "the predicate is derivability, not merely grain and node kind" — which restates rather than defines it), and *"where it applies but resolves poorly, it renders and states its own thinness"* has no bound for "poorly" or required form for "states." These are precisely the "handles X gracefully" pattern the rubric says to flag every time.

FR-1's new consequences are stronger but incomplete in one important way: they define honest states for *unparsed regions* and *unresolved dispatch targets*, and then commit to resolving `COPY`/`COPY … REPLACING` — without defining what happens when a copybook member cannot be found on disk or the search path is ambiguous. In a real COBOL estate that is not an edge case; it is the second thing that happens after cloning the repo. The PRD has just declared that not-knowing must always be visible and typed, and then leaves the most likely not-knowing untyped.

There is also an unresolved interaction between the new third honesty state and FR-4's structural invariant. FR-4 says "no source, no node." An unresolved-target element is by definition a node whose *target* has no source location. The UX spine resolves this (the anchor is the invocation site, shown in the popover); the PRD does not say so, so an implementer reading FR-4 and FR-1 together can reasonably conclude the `‹dynamic›` node cannot be emitted at all.

### Findings
- **high** Unresolved-target state vs. FR-4's "no source, no node" invariant is unreconciled (§3.1 FR-1, §3.2 FR-4) — FR-4 requires a source span per emitted element; the new unresolved-target node's defining property is an unknown target. `.memlog.md` line 25 says this decision "lands as a consequence on FR-1/FR-4" — it landed only on FR-1. *Fix:* add an FR-4 consequence stating that an unresolved-target element anchors on the *invocation site* span, and that this satisfies (not excepts) the provenance rule.
- **high** No honest state for an unresolvable copybook (§3.1 FR-1) — FR-1 commits to resolving `COPY`/`REPLACING` but defines no behaviour for a missing, ambiguous, or out-of-search-path member. Given the PRD's own doctrine, this needs a named state, not an implementer's guess. *Fix:* add a fourth consequence: an unresolvable `COPY` renders as a distinct state (neither an unparsed region nor a resolved inclusion) naming the member it could not find; extraction never silently proceeds as if the member were empty.
- **high** The "view is a stated snapshot / stale notice" trust rule did not make it into the PRD (§3.3 FR-5, §3.4 FR-6, cross-ref UJ-2) — the UX spine commits that a view does not silently re-derive when files change, that the header states staleness, and that a jump-to-source into a changed file says so. That is a *what must be true* truth claim (a stale view that looks fresh is exactly the lie this product exists to refuse), not a look-and-feel question, and it is load-bearing for UJ-2. The reconciliation imported copybook expansion and the honesty states but left this one behind. *Fix:* add an FR-5 or FR-6 consequence covering snapshot staleness and stale-anchor click-through.
- **medium** Two untestable FR-5 consequences (§3.3 FR-5) — "can honestly derive something" and "states its own thinness" carry no verifiable condition. *Fix:* give the availability rule a concrete predicate (e.g. "offered iff the perspective's required element set is non-empty for the current scope") and require the thinness statement to be a header-level count of resolved-vs-total (the UX spine's own "2 of 14 calls have static targets" is already the testable form — lift it).
- **medium** "Extraction snapshot" appears as a load-bearing noun with no definition (§3.3 FR-5, "Switching perspective re-projects the existing extraction snapshot: no re-parse, selection preserved") — FR-1 exposes "a queryable syntax representation"; whether that is the same artifact, and whether it is persisted, is an architecture-shaping question introduced only as a side clause. *Fix:* define it in §9 and tie it to FR-1's representation explicitly.

## 5. Scope honesty — thin

The struck-through OQ-3 and Assumptions entry are exactly right — the update preserves its own history rather than rewriting it. Non-Goals remain strong and specific.

But the update added scope without adding scope discipline. §5.4 still asserts "Every v1 exclusion is captured in §4 (Non-Goals) — nothing additional here," which the same update falsified: FR-5's new Out of Scope block excludes a bundled state-machine perspective (a genuinely new exclusion from the UX run) and "tool builds the tool," and neither appears in §4. The Assumptions Index gained no entries despite the update introducing several unconfirmed inferences — most notably that a one-perspective gate slice yields evidence that transfers to a six-perspective build (an explicit override of the UX spine's two-perspective "gate-sufficient" bundle), and that Phase B is inside v1. Open-items density is otherwise appropriate for the stakes.

Separately, §0 claims inferred content is "additionally tagged inline `[ASSUMPTION]`" where asserted at a specific point — there is not a single live inline tag in the document. The claim is hedged enough to be technically survivable, but the update is precisely the moment it stopped being true.

### Findings
- **high** §5.4's "nothing additional here" is now false (§5.4 vs §3.3 FR-5) — the state-machine non-bundling decision is a new, real v1 exclusion introduced by this update and lives only inside an FR's Out of Scope block. *Fix:* add the state-machine exclusion (and "tool builds the tool") to §4 Non-Goals, or reword §5.4 to point at both §4 and the per-FR Out of Scope blocks.
- **medium** Assumptions Index not updated for the update (§8) — no entry for the one-view gate slice's transferability, for Phase B's v1 membership, or for the assumption that resolving `COPY … REPLACING` is tractable within the base grammar + community extensions. *Fix:* add these three; tag the first inline at §5.2 since it is asserted at a specific point.
- **low** §0's inline-tagging convention claim has no live instances (§0) — no `[ASSUMPTION]`, `[NOTE FOR PM]`, or `[NON-GOAL for MVP]` tags exist in the body. *Fix:* either tag the new inferences above or drop the clause.

## 6. Downstream usability — thin

This PRD is chain-top — it feeds architecture and epics next, and the UX spine explicitly says "the PRD update should run before architecture consumes this pair." So this dimension matters more than usual, and it is where the update's cost is concentrated.

§5.2's Phase B row names seven deliverables. Four of them — **view gallery**, **view-trail history**, **provenance rail**, and **molded-tool perspective registration** — appear nowhere else in the PRD. There is no FR, no testable consequence, no Glossary entry. An epic writer source-extracting from this PRD would either drop them or invent requirements for them from a phasing table's middle column. Molded-tool perspective registration in particular is a *contract* capability that materially changes FR-3's surface, and FR-3 does not mention it.

The Glossary was not touched by the update at all. §3.3 introduces the entire viewing vocabulary — *scope*, *perspective*, *grain*, and the six perspective names — and §9 still defines only "Starter view," whose definition ("a small, v1 hardcoded set of view types (e.g. dependency map, call-flow)") is now doubly stale: it uses "view type" where the document has moved to "perspective," and it offers call-flow as an example of a day-one starter when §5.2 puts call-flow in Phase B.

Finally, "Phase A" now denotes two different bundles: in `EXPERIENCE.md` it is gate-sufficient and *includes* the unit flowchart; in `prd.md` §5.2 the flowchart is Phase A *remainder*, post-gate. §0's authority clause ("this PRD now governs" on scope) resolves it in principle, but only for a reader who reaches for §0 — an architect reading the two phasing tables side by side sees a contradiction, not a precedence rule.

### Findings
- **critical** Four §5.2 deliverables have no requirement backing anywhere in §3 (§5.2 Phase B) — view gallery, view-trail history, provenance rail, and molded-tool perspective registration exist only as line items in a phasing table. Epics derived from this PRD would build them from a table cell or not at all, and perspective registration silently expands FR-3's extension contract. This is the finding most likely to make downstream build the wrong thing. *Fix:* give each an FR (or an explicit consequence on FR-3/FR-5) with at least one testable consequence, or move them out of the PRD into the UX spine's ownership and say so in §5.2. Provenance rail and perspective registration deserve FRs; gallery and trail could be delegated with an explicit pointer.
- **high** Glossary missing the entire new viewing vocabulary (§9) — `scope`, `perspective`, `grain`, `extraction snapshot`, `honest state`, `dual anchor`, `copybook expansion`, `gate slice` are all used as load-bearing domain nouns and none is defined. The template's rule is explicit: a new domain noun in Features goes into the Glossary in the same pass. *Fix:* add entries, with `perspective` defined in terms of `scope` and `grain` and cardinality noted (six perspectives, three grains).
- **high** "Starter view" is now a stale synonym for "perspective" (§9, used at §3.1 FR-1, §3.3 title and FR-5 consequences) — the document uses both interchangeably, and the Glossary entry's own example (call-flow) is a Phase B item, implying availability the phasing denies. *Fix:* redefine "Starter view" as *a perspective bundled with the toolkit, as distinct from a molded tool* (this is the real distinction — see `addendum.md` on where the source lives), and drop the misleading example.
- **high** "Phase A" collides across PRD and UX spine (§5.2 vs `EXPERIENCE.md` § Views & Perspectives) — same label, different contents, both current, both "final." *Fix:* rename the PRD's tiers (e.g. "Gate slice / Day-one remainder / Post-gate") or add an explicit note in §5.2 that this supersedes the spine's Phase A/B split, and file a matching note in the spine.
- **medium** The Code perspective is unscheduled (§3.3 table vs §5.2) — six perspectives are committed; the phasing table assigns dependency map (gate), flowchart (Phase A remainder), and sequence/reach/call-flow (Phase B). The Code perspective appears only as "Code-perspective selection sync" in Phase B, leaving the base perspective (open the native editor split at the unit's span) in no tier. Since jump-to-source is already in the gate slice, this is probably Phase A — but it is a guess. *Fix:* assign it explicitly.
- **medium** `addendum.md`'s output model does not accommodate the Code perspective (`addendum.md` § v1 delivery shape, bullet 1) — it lists "code" among starter tools "pre-written … bundled inside the extension package" whose output is "rendered-view JSON" watched from a folder. The Code perspective renders nothing to that folder; it drives VS Code's native editor with bidirectional selection sync. The addendum was partially updated on 2026-07-19 (the asymmetry bullet) but this bullet was not. *Fix:* carve the Code perspective out of the JSON-output model in the addendum.

## 7. Shape fit — strong

Correctly shaped for a solo/hobby-rigor project with a public ship: light UJs in the single-sentence dial, operational rather than user-funnel SMs, NFRs deliberately deferred to architecture with the deferral indexed rather than silently dropped, and technical-how pushed to `addendum.md` instead of bloating §3. The update did not distort the shape — §5.2 is a legitimate adapt-in (Rollout and Change Management cluster) and earns its place. Moving the Glossary to §9 is a deliberate documented deviation from the template's §3 slot, not drift. No findings.
